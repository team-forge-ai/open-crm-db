# open-crm-db — genericization plan

## Goal

Turn `picardo-internal-db` into `open-crm-db`: a generic, MIT-licensed,
open-source-ready TypeScript/Postgres toolkit that any team can use as the
schema and migration backbone for a headless internal CRM / knowledge DB.

The repo is private for now and is intended to be made public after review.

## Acceptance criteria

1. No public-facing references to "Picardo", "PICARDO", "picardo" in source,
   docs, scripts, migrations, schema, env files, or skill (case-insensitive).
2. No references to "Linear's model" or "Linear-derived" framing in task
   schema docs / migration comments.
3. MIT `LICENSE` at repo root, attributed to team-forge-ai contributors.
4. `package.json` renamed to `@team-forge-ai/open-crm-db`, `license: "MIT"`,
   `private: true`, with a generic description.
5. CLI binary renamed `picardo-db` → `open-crm-db`. Package script alias
   `pnpm open-crm-db` works. `--help` output reads as a generic CRM DB tool.
6. Env vars renamed `PICARDO_DB_*` → `OPEN_CRM_DB_*`.
7. SQL trigger functions renamed `picardo_set_updated_at` →
   `crm_set_updated_at`, `picardo_search_text` → `crm_search_text`. All
   migrations + `schema.sql` updated consistently.
8. `.env.example` is generic, no private hosts/credentials/identifiers.
9. README, AGENTS.md, docs/ describe a generic open-source headless CRM DB.
10. Schema docs describe a generic task/work-item model with no Linear
    inspiration framing. Linear-specific *importer* removed (see scope cuts).
11. Local skill renamed `skills/picardo-internal-db` → `skills/open-crm-db`,
    with all internal references updated.
12. `pnpm typecheck`, `pnpm lint`, `pnpm test`, `pnpm build` all pass.
13. CLI `--help` exits 0 and shows the new branding.
14. No private/operational artifacts committed (no transcripts, no notes
    dumps, no real domains, no credentials).
15. Commit on `main`, push to `team-forge-ai/open-crm-db`. Repo stays private.

## Scope cuts (with rationale)

The following pieces were removed from the open-source release because they
are too tightly coupled to private operational workflows. They remain in the
git history of `picardo-internal-db` for the original team.

- **`enrich` command** (`src/commands/enrich.ts`): hardcoded healthcare
  organization categories (`provider_network`, `genomics`, `lab`, `imaging`,
  `hospital_system`, `payer`, `health_share`, `pharmacy`, `supplement`),
  `healthcareRelevance` and `likelyUseCasesForPicardo` enrichment fields, and
  Perplexity prompt fingerprint tied to the Picardo CRM. Removing it also
  drops `@ai-sdk/perplexity`, `ai`, and `zod` as direct dependencies.
- **`linear import` command** (`src/commands/import-linear.ts`): a 1100+ line
  Linear MCP importer that ships with hard assumptions about a private
  Linear workspace. Open-source users should not be steered toward Linear by
  default. Future work: a separate optional package.
- **`reflect-link-notes-last-six-months.md`**: 295 KB of internal notes,
  obviously private.

What we keep:

- `migrate` (up/down/status/create) and `info` commands.
- `embeddings backfill` command — generic MLX-on-Apple-Silicon embedding
  pipeline, no Picardo-specific schema knowledge.
- The full schema (organizations, people, affiliations, interactions,
  transcripts, AI notes, extracted facts, documents, tags, relationship
  edges, partnerships, semantic embeddings, sources, external identities,
  tasks/projects/teams).
- `scripts/export-graph.mjs` — visualization helper.

## Risks / caveats

- Migration filenames keep their original Unix-ms timestamps; only the file
  *content* is genericized. No users have applied these migrations yet, so
  rewriting content is safe. If anyone has already pointed the old CLI at a
  database, they would need to drop and re-run, but that is an acceptable
  trade for a brand-new open-source baseline.
- Renaming SQL functions (`picardo_set_updated_at` → `crm_set_updated_at`)
  in the initial migration *and* every migration that references them must
  be done atomically; otherwise `migrate up` against an empty DB will fail.
- Tests reference the old name in some fixtures (e.g. `picardo.health` in
  the enrichment tests). Since `enrich` is removed, those tests are removed
  with it.
- The schema dump file `schema.sql` is regenerated by `pg_dump` after a
  `migrate up`; we hand-edit it now so the committed snapshot matches the
  rebrand without requiring a live DB.
